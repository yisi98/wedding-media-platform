# API Design Guide

This document defines conventions for the Wedding Media Platform REST API. All endpoints must conform.

## Base URL & Versioning
- **Base**: `https://api.weddingmedia.example/api`
- **Versioning**: URL path versioning — `/api/v1/...`
- Breaking changes require a new version (`v2`); never break `v1` in place

## HTTP Methods
| Method | Purpose | Idempotent |
|--------|---------|------------|
| GET    | Retrieve resource(s) | Yes |
| POST   | Create resource / non-idempotent actions | No |
| PUT    | Full replacement | Yes |
| PATCH  | Partial update | No (typically) |
| DELETE | Remove resource | Yes |

## Resource Naming
- Plural nouns: `/events`, `/media`, `/users`
- Nested resources: `/events/{event_id}/media`
- Use `kebab-case` for multi-word paths (rare): `/access-codes`
- Use UUIDs in paths, not sequential IDs

## Standard Endpoints
```
GET    /api/v1/events              # List (paginated)
POST   /api/v1/events              # Create
GET    /api/v1/events/{id}         # Retrieve
PATCH  /api/v1/events/{id}         # Update
DELETE /api/v1/events/{id}         # Delete

GET    /api/v1/events/{id}/media   # List nested
POST   /api/v1/events/{id}/media   # Create nested
```

## Request Format
- Content-Type: `application/json` (except file uploads)
- All POST/PATCH/PUT bodies validated by Pydantic schemas
- File uploads use presigned URLs (see ADR-003), not multipart

## Response Format

### Success (200, 201)
```json
{
  "data": { ... } | [ ... ],
  "meta": { ... }
}
```

### List Responses (paginated)
```json
{
  "data": [ ... ],
  "meta": {
    "page": 1,
    "page_size": 20,
    "total": 142,
    "total_pages": 8
  },
  "links": {
    "self": "/api/v1/events?page=1",
    "next": "/api/v1/events?page=2",
    "prev": null
  }
}
```

### Error Response (4xx, 5xx)
```json
{
  "error": {
    "code": "RESOURCE_NOT_FOUND",
    "message": "Event with id 'abc-123' not found",
    "details": { "resource": "event", "id": "abc-123" },
    "request_id": "req_01HABC..."
  }
}
```

## Error Codes
Use both HTTP status codes and machine-readable error codes.

| HTTP | Error Code | When |
|------|-----------|------|
| 400 | `VALIDATION_ERROR` | Malformed request |
| 401 | `UNAUTHENTICATED` | Missing/invalid token |
| 403 | `FORBIDDEN` | Authenticated but not authorized |
| 404 | `RESOURCE_NOT_FOUND` | Resource doesn't exist |
| 409 | `CONFLICT` | Duplicate / state conflict |
| 422 | `UNPROCESSABLE` | Valid format but semantically wrong |
| 429 | `RATE_LIMITED` | Too many requests |
| 500 | `INTERNAL_ERROR` | Unexpected server error |
| 503 | `SERVICE_UNAVAILABLE` | Dependency down |

Never expose stack traces or internal details in error responses.

## Pagination
- Query params: `?page=1&page_size=20`
- Default page_size: 20, Max: 100
- For large datasets, support cursor pagination: `?cursor=xxx&limit=20`

## Filtering & Sorting
- Filters: `?status=active&created_after=2026-01-01`
- Sort: `?sort=-created_at` (prefix `-` for descending)
- Search: `?q=search+term`

## Authentication
- Bearer token in `Authorization` header: `Authorization: Bearer <jwt>`
- Refresh token in httpOnly cookie (see ADR-002)
- Public endpoints documented explicitly

## Headers

### Request
- `Authorization: Bearer <token>` — for authenticated endpoints
- `X-Request-ID: <uuid>` — optional client-supplied trace ID
- `Accept-Language: en` — for localized responses

### Response
- `X-Request-ID: <uuid>` — always returned
- `X-RateLimit-Limit: 100`
- `X-RateLimit-Remaining: 87`
- `X-RateLimit-Reset: 1700000000`

## Rate Limiting
- Default: 100 req/min per user
- Auth endpoints: 10 req/min per IP
- Upload endpoints: 30 req/min per user
- 429 response with `Retry-After` header

## Timestamps
- All timestamps in ISO 8601 UTC: `2026-05-24T15:30:00Z`
- Never use locale-dependent formats
- Use `_at` suffix: `created_at`, `updated_at`, `deleted_at`

## IDs
- All resource IDs are UUIDs v4
- Returned as strings in JSON
- Never expose sequential / guessable IDs

## Field Naming
- `snake_case` in JSON (matches Python backend)
- Frontend converts to `camelCase` at boundary if desired
- Boolean fields: `is_active`, `has_avatar`, `can_upload` (not `active`, `avatar`)

## Documentation
- OpenAPI spec auto-generated by FastAPI at `/api/v1/openapi.json`
- Swagger UI at `/api/v1/docs`
- Every endpoint must have:
  - Description
  - Request/response examples
  - All possible error responses listed
  - Required permissions documented

## Backwards Compatibility (within a version)
**Allowed:**
- Adding new optional fields to requests
- Adding new fields to responses
- Adding new endpoints
- Adding new optional query params

**Not allowed (requires new API version):**
- Removing fields from responses
- Changing field types
- Making optional fields required
- Changing endpoint paths
- Changing error code semantics

## Webhooks (Future)
- Signed with HMAC-SHA256
- Retry policy: exponential backoff, max 24h
- Idempotency via `X-Webhook-ID` header
